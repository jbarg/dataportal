Introduction

The Central Laboratory of the Research Councils (CLRC) in the UK is one of Europe's largest multidisciplinary research support organisations. From its three sites, CLRC operates several large scale scientific facilities for the UK research and industrial community including accelerators, lasers, telescopes, satellites and supercomputers which all create copious quantities of data. Currently CLRC is holding data in access of 50 TB, deployed through a number of data centres including 1 World Data Centre, 3 National Data Centres and a range of facilities for particular communities or instruments. However, it is expected that much more data will be collected in the future with the advent of new instruments, facilities (diamond) and projects (LHC Challenge), which will lead to data volumes in excess of several PB within the next 5-6 years. The data held at CLRC covers most major science areas e.g. Astronomy, Biology, Chemistry, Environmental Science and Physics.

Currently the scientists are forced to manually relate between all the experimental, data, computing and analysis facilities that are available world wide, with little infrastructure support (Figure1 Current situation).

  

Figure 1 Current situation

In the future it is hoped the Grid will provide these functions, enabling the scientists to choose much more easily from a wide range of services, connecting and combining desired services for an optimal working environment. Much of the access to the Grid is envisaged to take place through customisable, community oriented Portals (Figure 2 )

 

Fig. 2. The "One Stop Shop" for Scientists"

A range of projects are providing the initial building blocks of an integrated solution for users of experimental, computing and data facilities, demonstrating on a few selected examples how basic technologies can be used to build middleware components that support high level scientific grid applications (see Figure 3 ).

 

Fig. 3. Building Blocks for an Integrated Solution

Data will play a pivotal role in the success of Grid or e-Science developments. Virtually all envisaged applications will need to be able to draw from and deliver to the distributed heterogeneous information/data sources with a variety of contents. Hence three major challenges are posed: data accessibility, data transfer and management of personal data. Data accessibility implies the capability to locate information/data without prior knowledge of its physical location or the form in which its contents is described. Furthermore scientists, as well as applications, need to be able to combine results from different sources. Data transfer relates to the problem of large data volumes that need to be transferred across the Internet. Management of personal data is concerned with the growing distribution of data produced by scientists within a Grid environment, which required new ways of keeping track and moving data for single scientists and more importantly for research groups. CLRC's integrated data system includes the following components a DataPortal for high-level access to multidisciplinary data, linking to existing data catalogue systems. These catalogues include metadata as well as links to the data itself. The data itself is held in various storage resources from local disks, over databases to multi terabyte tertiary tape systems. To make the solution independent from the physical location(s) of the data we have introduced the Storage Resource Broker (SRB) from the San Diego Supercomputing Centre. The catalogues hold logical links into SRB, which SRB resolves into physical links to the data. To reduce and manipulate the data before shipping we have installed the RasDaMan software developed by Active Knowledge. See Figure 4 with an example for earth science (es) data.

A Walkthrough some key Features

In the following sections we describe most of the key interfaces that you will encounter while using the DataPortal and briefly explain the various functionalities in the process. A more detailed description of each interface can be found in section 4.

We use an example retrieving information from the BADC database that is used in the Estedi project.

Logging on

The initial page of the DataPortal can be reached from http://esc.dl.ac.uk:9000/dataportal and is shown in Figure 6 .

Figure 6 Welcome page

As a registered user you can log in from here by clicking on the Logon button. If you are not yet registered you need to contact the DataPortal team for a new account.

In the Login screen type in your user name and passphrase, if you are asked for them, to log in. The basic search function is presented to you with the option of doing an advanced search if you wish (see Figure 7 ).

Basic Search

This is a simple search, which uses some of the information in the central catalogue to narrow down the search to some relevant experimental data.

The search has three stages: first the facility and disciplines are selected, then the studies within those disciplines and finally the experiments within those studies.

Use the basic search to narrow down the search to particular data within studies in a discipline available within a specific facility.

Figure 7 Basic database search, prior to a selection

First select the facility from the drop down list. As you can see BADC is the initial selection. You can also choose additional facilities e.g. the MPIM in Hamburg, which hold similar data to the BADC and search both facilities at the same time. When choosing a discipline the search parameters will be automatically expanded providing more key words in the list to describe the exact discipline you are interested in (see Figure 8 ).

 

 

Figure 8 Basic database search - 3 levels of discipline selected

The disciplines you have chosen are displayed to the right of the discipline list. You have to choose four disciplines for the BADC facility in order to narrow your search sufficiently to be able to complete the basic search. You will know when you have chosen sufficient disciplines because the discipline box will only have one entry, ‘—Back –‘.

You can select another Facility or Discipline at any time from the pull down menu or list.

You have the option to limit the time taken for the query to return your results. The default time is 10 seconds and normally should be long enough. But sometimes, for example, through slow network connections this might take longer than expected. Therefore you can limiting the time for the query or click on no maximum which will return the results whenever all the information from each facility is returned.

When you have chosen four disciplines, the Submit button is displayed so that you can submit your query. This will then lead you to the next stage.

The next stage is Select a study name.

The final stage is Select the Experiments.

The Architecture

The DataPortal was designed to work in distributed and heterogeneous environments, our current test installation has proven that to be true, integrating a multitude of systems types, operating systems, databases and sites seamlessly. Furthermore the architecture of Version 2 of the DataPortal has proven to be robust. The current installation integrating data from three CLRC departments (SRD, ISIS and BADC) is shown in Figure 16 below.

 

Figure 16 DataPortal and connected services

In the following section we will describe the internal architecture of the Data Portal and how its different parts interact with each other. Central to the Data Portal is the Server, which provides the User interface and manages the interaction with the User and all attached resources. The server (see Figure 17 ) provides the user with a web interface to search the existing metadata both on the server itself and the connected data holdings transparently. Incoming requests from the user will be interpreted by the server and a query will be formed and transmitted either to the Central Metadata Catalogue or the connected local Metadata Repositories. Currently queries to local repositories have a SQL like notation and are transmitted via http. The result from the various local repositories and the Central Catalogue is expected in XML format. The response generator will collate the results and user output generator will produce the required pages to display the results. The Server is also responsible for the user authentication and session control.

 

Figure 17 Data Portal Server Architecture

CLRC has developed a special multidisciplinary metadata format to be able to integrate and make available data from various scientific topics ranging from astronomy to physics. General information about the type of data held within CLRC and connected sites are kept in the General Metadata Repository, however more detailed information needs to be extracted from local repositories. In the following paragraph we will describe how these resources are connected with the server (see also Figure 18 ).

Other repositories are expected to have either their own metadata catalogue systems and their own metadata formats describing their data holding or use an extension of the CLRC metadata format. To integrate them each catalogue will be accompanied by an XML wrapper, which interprets the requests of the Data Portal server and translates them into local SQL and metadata formats. If the reply consists solely of metadata information the XML output generator and the Response generator will provide a collated XML document to the Data Portal server, with the included metadata translated back into the general metadata format. Does the request also include the wish for actual data, these will be transferred separately and the XML document returned to the server will include the necessary information. Currently the metadata catalogues include hard links to the physical data location, this site will be contacted and the data transferred via http to the user. Figure 18 depicts the integration of the BADC data.

 

Figure 18 Integration of External Data Catalogues

The core DataPortal system offers the user the possibility to collect all relevant datasets in his personal shopping basket, which can be kept from one session to the next if required. This shopping basket than offers the user a range of functionalities like download (using http), delete (from shopping basket) or if available manipulate. Datasets have to be marked in their metadata database for the Portal to know a manipulation service is available. In the case of the BADC this manipulation is the link to RasDaMan. The user is guided to a manipulation interface, which interacts both with the metadata database (via the XML wrapper) for general information about the dataset as well as the Gribservelet, which enables the interaction with the RasDaMan database. Both the Gribservelet and the XML wrapper are implemented as Grid services (see Figure 19 ).

Retrieval of information about the data

The data sets are collected together into ‘studies’ within ‘disciplines’. For each study, you can retrieve all the data sets or individual data files.

Each data set comprises one or more data item that you can download to view or save.

You can find the data items by using the ‘basic search’ or the ‘advanced search’ facilities.

Immediately after logging in you are presented with the ‘basic database search’ page.

Using the basic database search

The basic database search enables you to select the data sets you are interested in by taking you through four simple steps:

    1. Select discipline and facility from the menu

    2. View the studies information and select some studies

    3. View the expanded information for the selected studies and select some data sets or data holdings

    4. View the list of data holdings, data sets and data files and click on the one you wish to download.

Each data item or study is associated with a discipline. A hierarchy of disciplines is stored by the Data Portal to help you narrow down your selection of studies.

Select the discipline and facility

Both facility and discipline have to be chosen for a successful search. You can either start with the discipline if you are unsure about suitable facilities (they will then be chosen automatically) or choose suitable facilities and you will be given a choice of valid discipline hierarchies.

Use the ‘Discipline’ pull down list to select from the available disciplines. You can also use it to move up or down a level in the hierarchy. At each level the number next to the discipline name shows the level of that discipline within the hierarchy.

Figure 22 Basic database search with four disciplines selected

When you are happy with your choices please press on the Submit button to initiate a search for the studies. If you press Submit before you have selected four levels of discipline you will get a reminder that four need to be selected and the search is not initiated.

For the BADC data we offer a RasDaMan interface (developed by the European ESTEDI project marketed by Active Knowledge). In the following we will describe the RasDaMan interface and functionalities.

RasDaMan

RasDaMan is a database management system specialising in raster data. The data in question is analysed and loaded into the database system. During the loading process the data is split up in suitably sized blobs (binary large object) and stored together with content relevant information. The user has than the possibility to query, manipulate and extract the inserted information based on his own requirements. The user can e.g. extract only selected information for example only temperature values for a certain area of the globe, but not all the other parameters stored in the file nor the information for other countries. Thus he can limit the finally extracted data dramatically. Furthermore mathematical manipulations and analysis can be carried out on the selected data e.g. max temperature for the UK on the 1/1/2000. The way to retrieve these information from RasDaMan is to use their own SQL like query language. However we have decided that it would be too inconvenient for most users to learn the language and have therefore provide an interface which gives access to some of the functionalities via simple menus.
