Information Index

Java Web Start is a helper application that gets associated with a Web browser. When a user clicks on a link that points to a special launch file (JNLP file), it causes the browser to launch Java Web Start, which then automatically downloads, caches, and runs the given Java Technology-based application. The entire process is typically completed without requiring any user interaction, except for the initial single click.

Once a application is downloaded the application can be run via 2 ways. Each way checks the web server where the application was oringinally downloaded for any application that has a later time stamp, and if it finds one will automatically download and update the software for patches and improvements etc.

  1. Re cliching the hyperlink on the page where you originally found the application.
  2. Open the Java Web Start Application Manager. On windows this is normally a item on your desk top for Programs menu. On linux execute $JAVA_HOME/jre/javaws/javaws or something similar. And if you click on 'View' and 'Downloaded Applications' you should see Data Portal MyProxy. This can now be launched by clicking 'Start'. The advantage of this is that the application can be run even if you are no cannected to the internet or the original web server is down.

Security Index

Applications launched with Java Web Start are, by default, run in a restricted environment where they have limited access to local computing resources, such as storage devices and the local network. In this sandbox environment, Java Web Start can guarantee that a downloaded and potentially untrusted application cannot compromise the security of the local files or the network.

An additional security feature supported by Java Web Start is digital code signing. If an application being invoked is delivered in one or more signed JAR files, Java Web Start will verify that the contents of the JAR file have not been modified since they were signed. If verification of a digital signature fails, Java Web Start will not run the application, since it may have been compromised by a third-party.

MyProxy | Retrieve Index

If the MyProxy | Configure has been set then this menu item should be enabled. This allows you to retrieve a set of delegated credentials from the MyProxy Server that you have configured. When requesting a set of proxy credentials from a myproxy, a set of credentials needs to be sent with the request for authorisation with the MyProxy Server that this person associated with the credentials that are been sent is allowed to retrieve proxies. Therefore you need to type in your Grid PassPhrase with this request to create a Proxy Credential to send to the MyProxy server to authorise this request.

 

Enter in the following and press 'Ok'

  1. Username, this is the username you used to sent a set of credentials to the MyProxy.
  2. MyProxy PassPhrase, this passphrase you used with the username when you sent a set of credentials to the MyProxy.
  3. Lifetime, this is the lifetime of the proxy that you are retrieving from the MyProxy.
  4. Grid PassPhrase, this is the password that you use to encrypt you $user.home/.globus/userkey.pem private key.

You Proxy credential should be retrieved from the MyProxy, and the details of the credential shown to you.

 

You now have the option of saving the proxy.

Tutorial Index

Before using the data portal or other tools you need to load your personal certificate into a proxy server. There are a range of proxy servers around the world that will hold your certificate accessible from anywhere in the world. The benefit is not just availability (you do not need to take your certificate with you when you travel a lot), but more importantly security as the proxy server will only pass on a subset of your credentials (proxy certificate) to the requesting services, therefore your credentials are safe. You can use our own proxy server to store your certificate available for upload under: http://tiber.dl.ac.uk:8080/myproxy/myproxy.jnlp.

At last a word of warning certificates uploaded into a MyProxy server have only a limited lifetime (the default is 7 days) for security reasons. So you will need to make sure that your certificate is still valid before you request a proxy from the MyProxy server the next time, otherwise you will need to re-upload your credentials.

Uploading your certificate into the proxy server

You will need to follow these instructions to upload your credentials if:

  • this is the first time you have used the portal or
  • you have been using the portal and your proxy lifetime has run out (usually after 7 days)

You will probably need to do this every week depending on the default settings of the MyProxy server. This server stores your certificate and issues a temporary or proxy certificate on your behalf to maintain the highest level of security within the portal applications.

You will know if you need to re-upload your certificate as you will receive an error message when you try to log on. Otherwise you can skip these steps and continue using the portal in the normal way.

Uploading your certificate

You can upload your certificate by going to the login page of the page at http://tiber.dl.ac.uk:8080/myproxy/myproxy.jnlp

 

 

Set up Java Cog Kit Index

This process usually only needs to be carried out when you first use the upload tool on your machine, please keep the resulting file in a directory where it cannot be removed. If you have installed the Cog Kit before please proceed to the next step 'Upload Proxy Certificate'. Otherwise carry out the following:

1.      Select Config | Setup CoG to open the Java Cog Kit configuration wizard. After reading the welcome page please click on Nextto progress the installation.

2.      Configure your user credentials. You will need your exported User Certificate and User Private Key. Either type the path of your exported certificate and key into the fields below the names or click on the '..' button to use the browsing facility to choose the correct file location (incl. file name). When finished select Next to go on to the next page.

 

 

3.      Download the certificate of your Certification Authority (CA) and register it with the Proxy tool by clicking on Download UK e-Science Certificate (if you have a certificate authorised by the UK e-Science Community) then press OK to proceed.

4.      You will than see a window that informs you of the successful download of the certificate giving location and name for the certificate, please press OK to proceed.

5.      From the next window 'Add CA Certificate', choose the certificate that you just downloaded by clicking on its name in the file list of the window. Press Select to proceed. You will than go back to the Configuring Certification Authorities (CA) page. If you are using another certification authority than the UK e-Science one please click on Add to add their certificate to the download tool list. This will open another window, which will allow you to browse your computer for the correct Certification Authority Certificate, once found please select it by clicking on the file name and press the Select button to add it to the download tools list of CA certificates. You will then go back to the Configuring Certification Authorities (CA) page. Please press Next to proceed to the next step.

6.      NOTE: You will need to add both the Root CA of your user certificate and the Root CA of the MyProxy server that you wish to upload the certificate to for the tool to work.

7.      The tool has now all the information it needs to create a temporary proxy file. The name and the location where the tool intends to place it will be shown to you on the Configuring Proxy File web page. Please check that you have write access to the chosen location, otherwise click on '..' to chose a more appropriate name or location. Press Select to confirm you new choice, this will take you back to the Configuring Proxy File page. Now click on Finish to complete the installation of the Java Cog Kit. A little window will appear to confirm the successful installation, please press OK to proceed. This will take you back to the Proxy Tool.

The Lifetime and strength can be left alone just kept as the normal default. If you wish to change them here are their meanings

  1. Proxy Lifetime. This is the maximum time that a set of Proxy Credentials can be pulled out of the MyProxy server. This is set to 2 hours by defualt for security reasons.
  2. Credential Lifetime. This is the maximum time that your set of credentials is valid for in the MyProxy Server. After this time has expired you will not beable to retrieve another set unless you upload another set of credentials. This is set to 1 week by default for security reasons.
  3. Strength. This is the strength of the public/private key that is uploaded into the MyProxy. This is set to 512 by default for security reasons

 

 

Then please press Ok to save your configuration details.

Why do i get a 'You have an old version of the jce security classes in your java classpath. Please remove $filename. All java applications need to be closed before it can be removed.' error?

Older versions of this application used jce-jdk13-117.jar security classes for the encrytion and decrytion of the credentials. This is used with the Globus Tool Kit in which this application is build upon. This file was signed with a invalid Java Sun Root certificate and therefore Java Web Start would not allow this file to be downloaded using Web Start. To get around this the file was manually downloaded and put into $JAVA_HOME/jre/lib/ext folder, where Web Start could use this file.

The problem now is that if this file exists in this folder this application will use these classes instead of the new security classes downloaded with this application, and therefore it will not work. Therefore the file that is prompted is needed to be removed. NB: since all java applications will be using all files in this directory you will not beable to remove it before all java applications are closed.back

I keep on getting error messages when i try and upload and download proxies from the MyProxy Server.
Errors:

"Username and credential in use by someone else". The user name that you have used for the MyPorxy server has already been used by another person.

"Unknown CA". Cog Kit set up incorrectly. The user and MyProxy Root CAs need to be added in the setup so that they can both authenticate each other.

"Can't get the local trusted CA certificate: Cannot find issuer certificate for local credential with subject". The MyProxy server is setup incorrectly and the user Root CA (issuer certificate) cannot be found to authenticate.
